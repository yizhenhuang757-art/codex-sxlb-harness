# 工作流参考

SXLB 将一项实质性请求转换为可见、可复核的路径。并非每个案卷都需要同等重量的全部工件，但案卷不应失去对两个问题的清晰回答：谁拥有下一项决策，以及什么证据支持它。

## 默认路径

```text
受理 → 提案 → 复核 → 派发 → 执行 → 完成复核 → 回奏
太子 → 中书省 → 门下省 → 尚书省 → 六部 → 门下复核 → 回奏
```

对于非常小的任务，SXLB 可以使用较轻路径：`太子 → 尚书省 → 单部执行 → 门下复核 → 回奏`。涉及规划、多个工具动作、文件修改、验证或实质风险的任务，通常使用完整路径。

## 1. 受理——太子

受理将请求立为案卷，判断任务等级，识别目标产出与约束，并将其路由到合适的路径。太子不会自行暗中完成规划，也不会自行跨部门拆分工作。

**有用的产出：** 简明的案卷说明、任务等级、范围、已知风险和下一责任机构。

## 2. 提案——中书省

提案阶段把案卷转化为可执行的方案。它会写明问题、预期结果、非目标、证据、验收标准、依赖项，以及需要复核的问题。不确定性应当出现在方案中，而不应藏在执行过程里。

**有用的产出：** 包含备选路径和明确复核问题的中书方案。

## 3. 复核——门下省

复核检查所提路径是否安全、适度、具有证据支持，以及是否真的具备执行条件。它可以批准、退回方案修订、要求补充证据，或将实质性分歧升级为有边界的会审。

**有用的产出：** 明确的裁决，以及继续推进所需满足的条件。

## 4. 派发——尚书省

方案获准后，派发将有边界的工作分配给负责部门；若真实子智能体确有明确理由，也可派发给它。派发包应写明负责人、可写范围、交付物、验证方法、回传格式和集成责任。

**有用的产出：** 派发记录，而不只是藏在对话中的一句指令。

## 5. 执行——六部

六个功能部门承担有边界的工作。SXLB 将其作为现代服务边界：工部负责实施，刑部负责测试与验证，礼部负责面向使用者的文档，户部负责数据与分析，兵部负责集成与自动化，吏部负责技能治理与框架维护。

这些部门不取代提案、复核或派发权。工作回传会记录做了什么、有什么证据，以及还存在哪些限制。

## 6. 完成复核——门下复核

完成是独立关口。复核检查执行回传、验证证据、未解决风险和所需记录。失败或不完整的结果会退回派发或执行，而不会默认被表述为已完成。

**有用的产出：** 说明是否可以结案的复核记录；若不可以，则说明下一步必须做什么。

## 7. 回奏与结案——回奏 / 退朝

最终回奏给出结果、证据、剩余风险和后续动作。结案会将可持续记录路由至合适的公共或用户自有位置，并将项目内学习与自动的全局系统改动分开。

## 何时适合多智能体工作

多智能体执行是可选项，而不是默认项。只有当分支确实相互独立、需要专门能力，并且并行的收益明显高于成本时才使用。派发之前，应评估候选能力、定义窄范围的工作包，并要求书面回传，使其能够被复核和整合。
