# 为什么还有 SXLB

SXLB 是面向 Codex 的独立实现。它受到 [cft0808/edict](https://github.com/cft0808/edict) 的启发，尤其是三省六部的智能体分工方式和其[任务派发架构](https://github.com/cft0808/edict/blob/main/docs/task-dispatch-architecture.md)。SXLB 不内置、不 fork，也不复刻 Edict 或 OpenClaw 的代码。

两者面对的是同一个问题：当多种能力都可能参与一项任务时，怎样让规划、复核、执行和责任归属保持可见？它们在不同环境中给出了不同答案。

## SXLB 从 Edict 借用了什么

Edict 让三省六部成为理解智能体分工的一套可用语言。SXLB 保留这套词汇，因为它让同一项任务不只保留一个视角。方案可以在行动前被复核，派发可以写明负责人，回传也可以在成为最终答案前接受检查。

这种关系是概念上的借鉴，并且有明确记录。它不表示代码共享、fork 关系，或相同的运行时架构。

## 两个项目有什么不同

| 问题 | Edict | SXLB |
| --- | --- | --- |
| 主要形态 | 多智能体编排系统 | Codex 技能工具 |
| 默认运行方式 | 由专门智能体协同完成工作 | 在一个对话中先走以文本为先的治理路径；派发是可选项 |
| 控制界面 | 智能体编排、面板、配置和审计式运行 | 案卷、可见的提案与复核关口、派发包、状态面板和可移植模板 |
| 派发 | 是编排的核心议题 | 是经过复核的选择：案卷须说明专业化是否必要，并给出有边界的工作包和可整合回传 |
| 公共包 | 项目运行时 | 经过脱敏的协议、技能、模板、脚本、测试和文档，不包含维护者的私有运行历史 |

这些区别不构成高下判断，只说明不同的取舍。

## SXLB 为什么采用这种形态

Codex 的工作常从一段人机对话开始。使用者需要看见正在提出什么，以及为什么这样安排。因此，SXLB 先提供一条较轻的路径：受理、必要时拟制方案、复核、有边界的执行、完成复核和回奏。当任务确有相互独立的分支，且协调成本值得承担时，才引入智能体或专门技能。

这种安排把三个常被混在一起的动作分开：决定做什么、实际执行，以及接受结果。它也把私有历史和凭据留在公共包之外。

## SXLB 为自己的使用场景做了什么调整

- 通过案卷和状态记录，把受理、复核、完成和重新进入任务的过程写明。
- 将外部智能体、技能和提示词先视为参考材料，再决定是否赋予实际职责。
- 真实派发必须有明确范围、交付物、验证方法和回传。
- 公共包可独立安装，不会带入维护者的个人数据或本地配置。

这些是为 Codex 技能工具作出的设计选择，并不表示历史机构、Edict 或其他系统应当以同样方式运行。
