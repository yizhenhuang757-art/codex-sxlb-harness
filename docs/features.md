# Outcomes and features

SXLB is useful when a task needs more than a chat transcript but less than a heavyweight project-management system. Its control surfaces make work inspectable without turning every request into bureaucracy.

## Expected outcomes

| Outcome | What SXLB provides |
| --- | --- |
| Clear next step | A case status and explicit owner for the next governed move |
| Reviewable decisions | Proposal, evidence, objections, and review gates instead of an unexplained handoff |
| Accountable execution | A bounded dispatch packet and recorded return |
| Durable context | Case records, event ledgers, and a status board rather than hidden conversational memory |
| Proportionate coordination | Optional multi-agent work only when specialization or parallelism earns its overhead |

## Core features

### Text-first governance chain

The default path is a clear legal chain: intake, proposal, review, execution, verification, and closure. The harness keeps human-readable records at each boundary.

### Status and recordkeeping

The status board reports current phase, open risks, approvals, and next action. Case packages, artifact registries, and event records preserve the evidence needed to revisit a decision.

### Review gates

An execution return is not silently treated as accepted. SXLB keeps separate space for objections, review, verification, and re-review when needed.

### Optional capability admission

External agents, skills, prompts, and patterns begin as reference material. They are evaluated and bounded before being appointed to active work; the companion skill helps decide when real subagents are worth the coordination cost.

### Portable public boundaries

The public edition ships templates, protocols, scripts, and tests. It does not ship private worklogs, memory stores, credentials, browser profiles, or personal tool inventories.
