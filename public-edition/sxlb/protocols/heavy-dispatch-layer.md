# Heavy Dispatch Layer

## Purpose

Define the cold-start "heavy weapon" layer for real multi-agent execution inside `sxlb`.

This layer is not the daily shape of `sxlb`. It is activated only after `尚书省` has a dispatch candidate that passes `sxlb-agent-dispatch-check`.

## Reference Boundary

OMX-style systems are useful here as execution-layer reference patterns:

- isolated worker environments
- durable branch state
- explicit task lifecycle
- structured worker returns
- verify and fix loops
- merge before final review
- hooks or logs as evidence capture

Do not absorb:

- OMX role identity or agent names
- always-on team mode
- UI or HUD assumptions
- command syntax as SXLB law
- any external dependency requirement

## Activation Gate

`尚书省` may activate this layer only when all conditions are true:

1. `sxlb-agent-dispatch-check` is `pass`.
2. `dispatch-order.md` marks `真实派发：yes`.
3. Every real branch has non-overlapping ownership or a named integrator for shared writes.
4. Every real branch has observable acceptance criteria and a return artifact.
5. `门下省` has approved the pre-dispatch package.

If any condition is missing, stay with local office execution.

## Heavy Layer Lifecycle

1. Admission: record `多 agent 准入`, cost judgment, and reason in `dispatch-order.md`.
2. Isolation: decide whether each worker needs a separate git worktree, clean workspace, or read-only research scope.
3. State: create a branch state record before workers begin.
4. Packets: issue `subagent-work-packet-<branch>.md` for each real branch.
5. Execution: workers operate only inside their packet.
6. Return: each worker writes `subagent-return-<branch>.md`.
7. Merge: a named integrator reconciles returns and writes `merge-summary.md` when outputs interact.
8. Verify/fix: `刑部` or the assigned verifier runs evidence checks; failed checks return to `尚书省` for recall, fix, or re-dispatch.
9. Review: `门下省` reviews dispatch, packets, returns, merge, and verification together.
10. Shutdown: record whether the heavy layer was closed, recalled, or paused; do not leave a fake active team after `回奏`.

## Branch State

Heavy dispatch should keep one durable branch state file in the case package:

- `heavy-dispatch-state.md` for human-readable state, or
- the existing `subagents/manifest.json` when generated by `subagent_dispatch.py`.

The state must track:

- branch id
- office owner
- execution environment
- write scope
- status: `queued`, `running`, `blocked`, `returned`, `merged`, `recalled`, or `closed`
- packet path
- return path
- verification state
- merge state
- last material update

This state is a checkpoint surface, not a dashboard promise.

## Isolation Policy

Use the lightest isolation that protects the work:

- read-only research branch: no worktree required
- docs-only branch: shared repo is acceptable if write scopes do not overlap
- code branch with possible conflicts: prefer separate worktree
- destructive or environment-changing branch: require explicit approval and stronger touched-file evidence

If isolation cannot be stated, do not dispatch real subagents.

## Return Contract

Every real worker return must include:

- what changed or was found
- touched files or artifacts
- commands or material actions
- verification evidence
- boundary statement
- unresolved risks
- requested next step

Claims without evidence are not review inputs.

## Verify/Fix Loop

Failed verification does not go straight to `回奏`.

`刑部` reports the failure, then `尚书省` chooses one:

- recall the branch
- re-dispatch to the same owner with a narrower packet
- split the branch
- downgrade to local execution
- ask the user for a HITL decision

Only after the loop closes may `门下省` allow final `回奏`.

## Closure Rule

At completion, the memorial should state:

- whether the heavy layer was activated
- which branches ran
- what state file or manifest recorded them
- whether all branches were returned, merged, recalled, or explicitly abandoned

If heavy dispatch was considered but not activated, record the cost judgment instead.
